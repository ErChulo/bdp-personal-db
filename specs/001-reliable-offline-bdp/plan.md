# Implementation Plan: Reliable Offline BDP

**Branch**: `N/A (existing repository; no branch hook configured)` | **Date**: 2026-06-22 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/001-reliable-offline-bdp/spec.md`

## Summary

Stabilize the existing React browser application instead of rewriting it. All SQL access will pass through a durable workspace coordinator that rehydrates exact IndexedDB bytes, serializes mutation in the worker, commits exported bytes in a strict fenced transaction, and reports success only after commit. Large import/backup work moves to streaming worker pipelines with staged validation. A browser-wide exclusive lease makes secondary tabs read-only, and service-worker updates wait for idle operations and explicit reload approval. Browser integration tests prove reload, offline, update, multi-tab, collision, and responsive journeys.

## Technical Context

**Language/Version**: TypeScript 5.9, React 19, Node.js 20 for build tooling  
**Primary Dependencies**: Vite 6.4, sql.js 1.14 (WASM worker), Dexie 4, Zustand 5, fflate 0.8; add Playwright test runner for browser contracts  
**Storage**: IndexedDB for SQL bytes, document collections, workspace metadata, ownership epochs, and staged operations; Cache Storage for versioned application assets; localStorage only for disposable UI preferences  
**Testing**: Vitest/jsdom for unit and adapter contracts; Playwright/Chromium for reload, multi-tab, offline, service-worker, responsive, and accessibility journeys  
**Target Platform**: Current Chromium-based desktop browser served over localhost or HTTPS; 360–1440 px layouts; production is not supported through `file://`  
**Project Type**: Static, client-only progressive web application  
**Performance Goals**: 10,000-row reports in under 1 second; visible progress within 250 ms for long work; no main-thread task over 100 ms during streamed import/restore; 500 MB fixtures complete without UI lockup  
**Constraints**: No runtime backend or external network; 500 MiB (`524,288,000` bytes) maximum per import/restore file; success follows durable commit; one writable tab; no silent empty SQL fallback; user data survives application updates  
**Scale/Scope**: One browser profile and workspace, multiple SQL databases and document collections, 11 primary sections, files through 500 MiB; reference environment is current Chromium on a 4-core machine with 8 GiB RAM and at least 1.5 GiB free origin storage

## Delivery Sequence

1. Establish workspace ownership, fencing, operation tracking, and read-only UI state.
2. Route SQL reads and writes through exact rehydration and atomic durable commit; add failure rollback and reload regressions.
3. Replace destructive existing-destination import with staged schema validation and transactional append.
4. Move import, backup, and restore parsing/compression to cancellable streaming workers; enforce the byte cap before reading.
5. Implement validated collision-safe restore and round-trip exports.
6. Replace immediate service-worker activation with waiting-worker messages, idle gating, confirmation, and controlled reload.
7. Add real-browser offline, multi-tab, update, navigation, responsive, accessibility, and 500 MB reference-environment verification.
8. Run `npm run typecheck`, `npm test`, `npm run build`, and the browser suite; confirm generated `dist/` matches source.
